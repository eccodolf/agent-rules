# Banking scenarios examples

## 1. Payment creation

Минимальные обязательные контроли:
- idempotency key;
- audit trail;
- transaction boundary;
- duplicate protection;
- controlled error handling.

Пример unsafe решения:
- принять запрос;
- сразу отправить во внешний провайдер;
- при timeout просто повторить запрос без проверки статуса.

Почему это плохо:
- возможен повторный side effect;
- нет reconciliation path;
- нет контроля duplicate processing.

Пример safe решения:
- сохранить входящий запрос и idempotency key;
- создать внутренний статус `pending`;
- записать audit event;
- выполнить внешний вызов через контролируемый async flow;
- при retry сначала проверять существующее состояние операции.

## 2. Anti-fraud rule execution

Что можно логировать:
- rule set version;
- decision id;
- итоговый outcome;
- безопасный идентификатор операции.

Что нельзя логировать:
- внутренние пороги;
- скоринговые коэффициенты;
- внутренние правила коммерческой тайны;
- персональные данные клиента в явном виде.

## 3. Manual approval / dual control

Что должно быть в дизайне:
- кто инициировал операцию;
- кто подтвердил операцию;
- когда были оба события;
- какой объект и какой статус изменился.

Что нельзя делать:
- позволять одному и тому же пользователю инициировать и подтверждать критичную операцию без явного исключения по политике.
