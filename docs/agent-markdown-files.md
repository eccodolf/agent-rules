# Использование Markdown-файлов в проектах для управления агентами

## Назначение

Markdown-файлы в репозитории проекта могут использоваться как управляемый и проверяемый слой инструкций для локальных AI-агентов. Такой подход позволяет хранить правила рядом с кодом, версионировать их, согласовывать изменения через pull request и применять их одинаково для всей команды.

Этот документ описывает, как использовать `.md`-файлы для управления поведением агента, не смешивая эти правила с бизнес-кодом и не создавая неконтролируемых источников инструкций.

---

## Зачем использовать Markdown-файлы

Преимущества:
- правила агента находятся в том же жизненном цикле изменений, что и код;
- все инструкции можно ревьюить и версионировать;
- проще отделять проектные правила от общих платформенных;
- можно задавать разные уровни инструкций: глобальные, проектные, модульные;
- удобнее ограничивать контекст, если агент читает только явно разрешенные файлы.

Markdown особенно полезен для:
- coding guidelines;
- project conventions;
- архитектурных ограничений;
- правил генерации кода;
- контрактов по форматам ответа;
- runbook и operational hints;
- ограничений по безопасности.

---

## Основные принципы

1. Markdown-файлы с правилами должны быть **явными**, **краткими** и **проверяемыми**.
2. Агент не должен автоматически читать все `.md` в репозитории без ограничений.
3. Должен существовать явный перечень директорий и файлов, которые считаются источниками инструкций.
4. Правила в `.md` не должны содержать секреты, токены, пароли, реальные ключи или чувствительные данные.
5. Любые проектные правила должны быть подчинены корпоративным и платформенным ограничениям ИБ.
6. Необходимо фиксировать приоритет между уровнями инструкций.

---

## Рекомендуемая структура

Один из рекомендуемых вариантов:

```text
/
  README.md
  AGENTS.md
  .agentignore
  docs/
    architecture/
    standards/
    runbooks/
  .ai/
    system/
      platform-rules.md
      secure-development.md
      prompt-templates.md
    project/
      coding-rules.md
      domain-context.md
      testing-rules.md
    modules/
      payments.md
      integrations.md
      reporting.md
```

### Назначение уровней

- `AGENTS.md` — краткая точка входа для агента в конкретном репозитории;
- `.ai/system/` — платформенные и обязательные правила;
- `.ai/project/` — проектные правила;
- `.ai/modules/` — правила для отдельных подсистем;
- `docs/` — расширенная документация, на которую агент может ссылаться по необходимости;
- `.agentignore` — список исключений для индексации и передачи в контекст.

---

## Рекомендуемые типы файлов

### 1. Входной файл для агента

Например: `AGENTS.md`

Должен содержать:
- краткое описание проекта;
- перечень основных директорий;
- правила выбора источников контекста;
- приоритет инструкций;
- запреты;
- ссылки на дополнительные `.md`-файлы.

### 2. Файлы с обязательными правилами

Например:
- `secure-development.md`
- `security-boundaries.md`
- `coding-rules.md`
- `testing-rules.md`

Должны содержать только устойчивые, регулярно применяемые правила.

### 3. Файлы с контекстом предметной области

Например:
- `domain-context.md`
- `business-rules.md`
- `integration-contracts.md`

Важно: такой контекст должен регулярно пересматриваться и не должен дублировать исходный код.

### 4. Шаблоны промптов

Например:
- `prompt-templates.md`
- `review-templates.md`
- `bugfix-templates.md`

Эти файлы нужны для унификации постановки задач агенту.

---

## Приоритет правил

Рекомендуется использовать следующий порядок приоритета:

1. Платформенные ограничения ИБ и системные правила;
2. Глобальные правила организации;
3. Правила конкретного репозитория (`AGENTS.md`);
4. Правила конкретного модуля или директории;
5. Задача пользователя в текущем запросе.

При конфликте побеждает более высокий уровень.

Пример:
- если пользователь просит изменить стиль логирования, но `coding-rules.md` запрещает такой формат, агент должен соблюдать проектное правило;
- если модульный файл допускает действие, но системное правило ИБ его запрещает, применяется системное ограничение.

---

## Что хорошо выносить в `.md`

Хорошо подходят:
- соглашения по именованию;
- стиль API;
- требования к тестам;
- принципы обработки ошибок;
- обязательные проверки безопасности;
- особенности предметной области;
- ограничения по зависимостям;
- правила миграций и работы с БД;
- формат сообщений коммитов и PR;
- запретные практики.

---

## Что не следует выносить в `.md`

Не следует размещать:
- пароли, токены, сертификаты, приватные ключи;
- реальные учетные данные;
- чувствительные клиентские данные;
- неактуальные инструкции;
- противоречащие друг другу версии правил;
- длинные неструктурированные описания без явных правил;
- слишком общие фразы, которые нельзя проверить.

---

## Рекомендации по формату

### Пишите правила в операционном стиле

Плохо:

```md
Нужно писать хороший код.
```

Хорошо:

```md
Всегда добавляй unit-тесты для новой бизнес-логики.
Не используй `print` для логирования.
Для ошибок внешних интеграций используй типизированные исключения.
```

### Делайте правила атомарными

Одно правило — одна проверяемая мысль.

### Явно указывайте область применения

Пример:

```md
Применяется только к директории `src/integrations/`.
```

### Добавляйте примеры

Если правило неоднозначно, приводите короткий пример «как правильно» и «как неправильно».

---

## Связь с агентным контекстом

Агенту рекомендуется читать `.md`-файлы не «все подряд», а по этапам:

1. прочитать `AGENTS.md`;
2. определить релевантный модуль;
3. подключить только нужные файлы из `.ai/project/` и `.ai/modules/`;
4. не включать нерелевантные документы в prompt.

Это критично для контроля объема контекста.

---

## Размещение правил рядом с кодом

Допустимо использовать локальные правила в директориях модулей.

Пример:

```text
src/
  integrations/
    AGENTS.md
    payments/
      AGENTS.md
```

Такой подход полезен, если:
- разные подсистемы имеют разные ограничения;
- интеграции сильно различаются;
- нужно локализовать контекст.

Но важно не допускать хаоса. Для этого:
- количество локальных файлов должно быть ограничено;
- правила должны наследоваться предсказуемо;
- все локальные инструкции должны быть перечислены в корневом `AGENTS.md`.

---

## Процесс управления изменениями

Изменения в agent-rules-файлах должны проходить так же, как изменения кода:
- через pull request;
- с обязательным review;
- с указанием причины изменения;
- с оценкой влияния на безопасность и качество генерации;
- при необходимости — с согласованием ИБ или архитектора.

Рекомендуется в PR указывать:
- какое поведение агента меняется;
- для каких задач применяется изменение;
- какие риски устраняет или создает новая инструкция.

---

## Проверка качества agent-rules-файлов

Нужно регулярно проверять:
- нет ли дублирующих или конфликтующих правил;
- нет ли неактуальных инструкций;
- не увеличивают ли файлы контекст без практической пользы;
- можно ли формализовать правило лучше;
- нет ли в документах чувствительных данных.

---

## Антипаттерны

### Плохо

- один огромный `AGENTS.md` на весь репозиторий без структуры;
- десятки локальных `.md` без единой точки входа;
- дублирование правил в нескольких местах;
- хранение исторических, уже неактуальных рекомендаций рядом с актуальными;
- включение в `.md` длинных дампов логов и необработанных обсуждений;
- отсутствие понятного порядка приоритета.

### Особенно опасно

- неявное разрешение агенту читать все `docs/` как trusted instructions;
- смешивание нормативных правил и справочных заметок без маркировки;
- хранение ограничений ИБ только в проектных файлах без системного enforce;
- попытка через `.md` обойти системные политики агента.

---

## Рекомендуемый минимальный набор для проекта

Минимум:
- `AGENTS.md`
- `.agentignore`
- `.ai/project/coding-rules.md`
- `.ai/project/security-rules.md`
- `.ai/project/prompt-templates.md`

Для крупных проектов дополнительно:
- `.ai/modules/<module>.md`
- `.ai/project/domain-context.md`
- `.ai/project/testing-rules.md`
- `.ai/project/review-checklist.md`

---

## Пример содержимого AGENTS.md

```md
# AGENTS.md

## Назначение
Этот репозиторий содержит сервис обработки заявок.

## Источники правил
1. `.ai/system/platform-rules.md`
2. `.ai/project/coding-rules.md`
3. `.ai/project/security-rules.md`
4. `.ai/modules/payments.md` — только для модуля платежей

## Ограничения
- Не изменяй публичные API без явного указания.
- Не добавляй новые зависимости без обоснования.
- Всегда добавляй тесты для новой бизнес-логики.
- Для SQL используй только параметризованные запросы.

## Как выбирать контекст
- Сначала читай этот файл.
- Затем подключай только релевантный модульный файл.
- Не включай в контекст содержимое `logs/`, `tmp/`, `dist/`, `node_modules/`.
```

---

## Вывод

Markdown-файлы — удобный способ управлять проектным поведением агента, если:
- есть единая точка входа;
- определен приоритет правил;
- есть ограничения на чтение и индексацию;
- правила версионируются и ревьюятся;
- они не подменяют системные меры ИБ, а дополняют их.
