# Архитектура локального AI-решения

## Назначение

Этот документ описывает базовую архитектуру локального AI-агента в закрытом контуре и границы доверия между пользователем, моделью, контекстом, инструментами и целевыми системами.

Цель:
- изолировать модель от неконтролируемого доступа;
- ограничить влияние агента на инфраструктуру;
- обеспечить проверяемость и воспроизводимость действий;
- разделить trusted и untrusted зоны.

---

## Основные компоненты

1. **Пользовательский интерфейс**
2. **Оркестратор агента**
3. **LLM / локальная модель**
4. **Контекстный слой**
5. **Policy engine и runtime guardrails**
6. **Инструменты и интеграции**
7. **Аудит, логи, метрики**

---

## Логическая схема

```text
User
  -> Agent UI
  -> Orchestrator
  -> Policy / Guardrails
  -> Context Builder
  -> Local LLM
  -> Output Filter
  -> Tools / Human Review / Result
```

---

## Границы доверия

### Trusted zone

- `.ai/system/*`
- утвержденные project rules
- allow-list инструментов
- policy engine
- audit pipeline

### Restricted zone

- исходный код проекта
- конфигурации
- документация
- тестовые данные

### Untrusted zone

- пользовательский свободный ввод;
- внешние файлы без валидации;
- логи, дампы, произвольные артефакты;
- ответы модели до post-check.

---

## Принципы архитектуры

- модель не должна иметь прямой неограниченный доступ к файловой системе;
- любой контекст должен проходить отбор и фильтрацию;
- mutating actions должны быть отделены от генерации текста;
- критичные операции требуют отдельного approval path;
- все значимые действия должны быть трассируемы.

---

## Контур выполнения

### 1. Вход

- запрос пользователя;
- идентификатор сессии;
- роль и разрешения;
- минимально достаточный контекст.

### 2. Подготовка контекста

- выбор релевантных файлов;
- удаление секретов и лишних данных;
- сокращение размера контекста;
- маркировка trusted instructions.

### 3. Генерация

- нормализованный prompt;
- системные ограничения;
- project-specific rules;
- runtime checks во время ответа.

### 4. Постобработка

- output validation;
- policy checks;
- redaction;
- решение: `allow`, `block`, `require_human_review`.

---

## Инструментальный доступ

Инструменты должны быть разделены по классам:

- `read_only`
- `safe_write`
- `privileged`
- `production_critical`

Для каждого класса должны быть определены:
- допустимые команды;
- условия вызова;
- требования к журналированию;
- необходимость human approval.

---

## Банковские требования к архитектуре

Обязательно учитывать:
- сегментацию среды разработки и production;
- раздельные роли для разработки, эксплуатации и ИБ;
- невозможность чтения секретов моделью по умолчанию;
- обязательный audit trail для критичных действий;
- dual control для чувствительных изменений.

---

## Минимальная production-схема

- локальная модель в изолированном контуре;
- context builder с allow-list;
- policy engine;
- runtime guardrails;
- журналирование запросов, решений и действий;
- отдельный approval-механизм для mutating operations.

---

## Вывод

Архитектура локального AI-решения должна исходить из того, что модель является недетерминированным компонентом с ограниченным доверием. Надежность достигается не одной моделью, а сочетанием изоляции, контекстных ограничений, policy enforcement и audit trail.
