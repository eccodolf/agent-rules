# Правила безопасной разработки при использовании модели

## Назначение

Этот документ описывает требования secure development для случаев, когда AI-агент участвует в генерации кода, конфигураций, SQL, инфраструктурных шаблонов и эксплуатационных инструкций.

Цель:
- снизить риск уязвимостей в сгенерированном результате;
- запретить опасные паттерны по умолчанию;
- ввести обязательную проверку перед применением изменений.

---

## Основные принципы

- любой output модели считается непроверенным;
- generated code не должен попадать в production без review;
- безопасность важнее скорости генерации;
- изменения должны быть обратимыми;
- критичные решения должны быть подтверждены человеком.

---

## Требования к коду

Запрещено:
- hardcoded credentials;
- отключение TLS, auth, audit;
- небезопасная десериализация;
- shell injection;
- SQL без параметризации;
- обработка ошибок с утечкой чувствительных данных.

Требуется:
- явная валидация входных данных;
- принцип least privilege;
- корректная обработка ошибок;
- журналирование security-relevant событий;
- покрытие тестами для критичной логики.

---

## Требования к API

- валидировать входные параметры;
- не раскрывать внутренние детали реализации;
- не возвращать секреты и лишние поля;
- использовать idempotency для финансово значимых операций;
- документировать ограничения и коды ошибок.

---

## Требования к SQL и БД

- только параметризованные запросы;
- запрет DDL/DML без явного подтверждения для production;
- миграции должны быть обратимыми;
- изменения схемы должны учитывать rollback;
- доступ к данным должен быть минимально необходимым.

---

## Требования к инфраструктуре

- не генерировать небезопасные default-конфигурации;
- не отключать контроль доступа и сетевые ограничения;
- не использовать latest tags без фиксации версии;
- явно задавать ресурсы, probes и политики рестарта;
- отделять dev, stage и production настройки.

---

## Требования к секретам

- секреты не хранятся в коде, конфиге и prompt;
- для секретов используется secret manager;
- примеры должны содержать только фиктивные значения;
- любые найденные секретоподобные строки подлежат redaction.

---

## Review и контроль изменений

Обязательно:
- code review человеком;
- статический анализ;
- тестирование;
- проверка соответствия policy engine;
- отдельная проверка для security-sensitive изменений.

---

## Минимальный secure-development checklist

- нет hardcoded secrets;
- нет опасных shell-команд;
- есть валидация входа;
- есть audit/logging для критичных действий;
- есть rollback plan;
- есть тесты или явное указание на их отсутствие.

---

## Банковские сценарии

Для платежей, антифрода, лимитов и подтверждений дополнительно обязательны:
- идемпотентность;
- audit trail;
- ролевая модель;
- retry-safe обработка;
- dual control там, где есть финансовый риск.

---

## Вывод

Использование модели не отменяет secure development lifecycle. Напротив, сгенерированный результат должен проходить более строгую проверку, чем обычной ручной код, потому что он может выглядеть правдоподобно при наличии критических дефектов.
