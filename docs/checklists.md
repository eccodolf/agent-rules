# Практические чек-листы для повседневной работы

## Назначение

Короткие списки контроля для пользователей, разработчиков, DevOps и ИБ перед использованием локального AI-агента и перед применением его результатов.

---

## 1. Перед отправкой запроса в модель

- задача сформулирована конкретно;
- указан ожидаемый формат ответа;
- добавлен только релевантный контекст;
- секреты и PII удалены или замаскированы;
- указаны ограничения и допущения.

---

## 2. Перед передачей файлов в контекст

- файлы действительно нужны для задачи;
- в контекст не попали `.env`, ключи, дампы, бинарные артефакты;
- большие файлы сокращены до релевантных фрагментов;
- trusted и untrusted источники различимы;
- объем контекста разумно ограничен.

---

## 3. После получения ответа модели

- ответ соответствует поставленной задаче;
- нет явных галлюцинаций;
- нет секретов, токенов, PII;
- соблюден формат ответа;
- учтены ограничения безопасности.

---

## 4. Перед применением сгенерированного кода

- код reviewed человеком;
- нет hardcoded secrets;
- есть валидация входа;
- нет опасных SQL и shell-паттернов;
- тесты выполнены или явно запланированы;
- понятен rollback-путь.

---

## 5. Перед применением инфраструктурных изменений

- изменение относится к нужной среде;
- нет скрытых production-side effects;
- есть approval для high-risk действия;
- включены журналирование и наблюдаемость;
- есть rollback plan;
- изменение не отключает защитные контроли.

---

## 6. Для банковских сценариев

- есть audit trail;
- обеспечена идемпотентность;
- retry не создает дублей;
- соблюдена ролевая модель;
- там, где нужно, предусмотрен dual control.

---

## 7. Для сопровождающей команды

- версия модели зафиксирована;
- версия правил и prompt templates зафиксирована;
- включены policy engine и guardrails;
- настроены метрики и алерты;
- обновления проходят через change management.

---

## 8. Минимальный stop-list

Остановить использование результата, если:
- модель просит передать секрет;
- ответ содержит credentials или приватные ключи;
- предлагается destructive action без approval;
- пропущены обязательные банковские контроли;
- нет возможности проверить результат.

---

## Вывод

Чек-листы нужны для того, чтобы снизить число ошибок в повседневной работе. Они не заменяют policy engine и review, но позволяют быстро остановить небезопасный или невалидный сценарий до его применения.
