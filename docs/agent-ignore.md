# Файлы игнорирования контекста агента (.*ignore)

## Назначение

Файлы вида `.agentignore`, `.aiignore`, `.llmignore` используются для управления тем, какие файлы и директории:
- НЕ должны попадать в контекст модели;
- НЕ должны индексироваться агентом;
- НЕ должны использоваться как источник инструкций.

Это критически важно для:
- защиты чувствительных данных;
- ограничения размера контекста;
- предотвращения prompt injection через файлы проекта;
- повышения качества ответов модели.

---

## Основные принципы

1. Все, что не нужно для решения задачи — должно быть исключено.
2. Любые чувствительные данные должны быть явно исключены.
3. Игнорирование должно быть строгим и предсказуемым.
4. Лучше исключить лишнее, чем случайно передать лишние данные в модель.

---

## Рекомендуемое имя файла

Предпочтительно:
- `.agentignore`

Допустимо (в зависимости от платформы):
- `.aiignore`
- `.llmignore`

Важно: использовать единый стандарт внутри организации.

---

## Базовый шаблон

```text
# Системные директории
.git/
.gitignore

# Зависимости
node_modules/
venv/
.env/

# Сборка и артефакты
dist/
build/
*.log

# Временные файлы
tmp/
cache/
*.tmp

# Секреты
.env
.env.*
secrets/
keys/
*.pem
*.key

# Базы данных
*.db
*.sqlite

# Логи
logs/
*.log

# Бинарные файлы
*.zip
*.tar
*.gz
*.bin

# IDE
.idea/
.vscode/

# Документация, не предназначенная для агента
internal_docs/
```

## Расширенные правила

### Исключение чувствительных данных

Обязательно исключать:
- конфигурации с паролями;
- файлы с токенами;
- дампы баз данных;
- экспортированные отчеты;
- ключи и сертификаты.

---

### Исключение шумных данных

Исключать:
- логи;
- временные файлы;
- артефакты сборки;
- бинарные файлы;
- большие дампы.

---

### Исключение нерелевантного контента

Исключать:
- старые версии документации;
- архивные директории;
- тестовые данные, не относящиеся к задаче;
- автоматически сгенерированные файлы.

---

## Белые списки (allow-list)

Вместо blacklist можно использовать allow-list подход:

```text
# Разрешаем только
!src/
!tests/
!AGENTS.md
!.ai/

# Остальное запрещено
*
```

Этот подход более безопасен, но требует аккуратной настройки.

---

## Связь с Markdown-файлами правил

Важно:
- `.agentignore` должен **разрешать только нужные `.md`**;
- не все `.md` файлы должны быть доступны агенту;
- правила (`.ai/`) должны быть явно включены.

Пример:

```text
# Разрешаем правила
!.ai/
!AGENTS.md

# Запрещаем все остальное
*
```

---

## Риски при отсутствии ignore-файла

- утечка чувствительных данных в prompt;
- превышение лимита контекста;
- ухудшение качества ответов;
- влияние мусорных данных на генерацию;
- prompt injection через файлы проекта.

---

## Интеграция с агентом

Агент должен:
1. читать `.agentignore` перед формированием контекста;
2. исключать файлы до индексации;
3. не обходить ignore-правила;
4. логировать, какие файлы попали в контекст.

---

## Антипаттерны

- отсутствие `.agentignore`;
- слишком общий allow-list без ограничений;
- игнорирование только `.gitignore` (этого недостаточно);
- включение `.env` в контекст;
- попытка передавать "всё, пусть модель разберётся".

---

## Рекомендуемый минимум

Каждый проект с агентом должен иметь:
- `.agentignore`
- `AGENTS.md`
- директорию `.ai/` с правилами

---

## Вывод

Файл ignore — один из ключевых механизмов безопасности при работе с LLM.

Он одновременно:
- ограничивает контекст;
- защищает данные;
- повышает качество ответов;
- делает поведение агента предсказуемым.
