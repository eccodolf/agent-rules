# Системные правила безопасности (обязательные)

## Критические запреты

Запрещено:
- выводить, вставлять или использовать секреты: пароли, токены, ключи, сертификаты, private keys;
- генерировать код с hardcoded credentials;
- предлагать отключение аутентификации, авторизации, audit logging или TLS;
- генерировать небезопасные SQL-запросы без параметризации;
- использовать небезопасные протоколы и конфигурации по умолчанию;
- предлагать destructive shell-команды без явного предупреждения и approval;
- предлагать отключение rate limit, validation, access checks и security headers ради "простоты".

## Работа с данными

- Использовать только обезличенные данные.
- Никогда не включать реальные данные клиентов, платежные реквизиты и персональные данные без явной необходимости и маскирования.
- Маскировать чувствительные поля в примерах, тестах и логах.
- Не использовать production-данные как учебный пример для prompt.

## Банковская и коммерческая тайна

- Не передавать в модель сведения, относящиеся к банковской тайне, если это не требуется для задачи и не прошло предварительное обезличивание.
- Не передавать в модель сведения, составляющие коммерческую тайну: внутренние тарифы, условия договоров, финансовые показатели, схемы антифрода, внутренние лимиты, служебные архитектурные детали и операционные метрики без явного допуска.
- Любые данные банковской или коммерческой тайны должны рассматриваться как чувствительные даже при работе в закрытом контуре.
- Если без таких данных задачу можно решить по интерфейсам, схемам и маскированным примерам, нужно использовать именно этот путь.

## Безопасная разработка

- Любой внешний ввод считать недоверенным.
- Входные данные валидировать на границах системы.
- SQL делать только параметризованным.
- Для файлов, URL и shell-параметров применять allow-list и нормализацию.
- Любые privileged actions должны быть отделены от обычной бизнес-логики.

## Интеграции

- Все внешние вызовы должны быть через защищенные каналы.
- Использовать таймауты, retry с ограничением и понятную обработку ошибок.
- Не повторять небезопасно неидемпотентные операции.
- Логировать ошибки без утечки чувствительных данных.

## Vault и секреты

- Все секреты брать только из Vault или утвержденного secret manager.
- Не хранить секреты в коде, репозитории, конфиге, примерах и prompt-ах.
- Не выводить значения секретов даже для "отладки".
- Не использовать реальные env values в текстовом ответе модели.

## DevOps и инфраструктура

- Не изменять инфраструктуру без явного запроса.
- Не генерировать destructive команды без предупреждения.
- Не предлагать изменения в production без rollback и post-checks.
- Не использовать `latest` и плавающие версии для критичных компонентов.
- Не ослаблять network policies, RBAC и securityContext без обоснования.

## Обязательные security-проверки в output

Любой output считается недоверенным и должен проходить проверку на:
- наличие секретов и персональных данных;
- наличие сведений, относящихся к банковской или коммерческой тайне;
- отключение security-контролей;
- небезопасный SQL и shell;
- отсутствие authz/authn в protected flows;
- отсутствие audit trail и idempotency там, где это критично.

## Когда нужен human review

Обязателен human review для:
- production-инфраструктуры;
- доступа к секретам;
- auth и permission changes;
- финансовых и банковских сценариев;
- SQL write, schema change и destructive maintenance operations.
