# Output rules

## Назначение

Контроль качества и безопасности output LLM перед применением результата в коде, конфигурации, инфраструктуре и review-процессах.

## Основные требования

- Не выводить секреты и персональные данные.
- Не выводить сведения, относящиеся к банковской тайне или коммерческой тайне.
- Не генерировать небезопасный код, SQL и shell.
- Не скрывать риски, побочные эффекты и ограничения.
- Не предлагать решение, которое нельзя безопасно проверить.
- Для production-sensitive изменений указывать проверки до и после применения.

## Что должен содержать хороший output

- решение именно поставленной задачи;
- соблюдение ограничений из prompt и `.ai`;
- безопасные defaults;
- минимально необходимый объем изменений;
- указание на тесты, валидацию и проверку rollback, если это нужно.

## Обязательные проверки

- SQL должен быть параметризован.
- Код должен быть безопасен на границах системы.
- Для banking нужны audit trail, idempotency и controlled failures.
- Для API нужны validation, предсказуемые ошибки и отсутствие утечки внутренних деталей.
- Для infra нужны version pinning, rollback и observability.

## Запрещено

- hardcoded credentials;
- раскрытие внутренних лимитов, anti-fraud logic, служебных схем принятия решений, внутренних финансовых метрик и договорных условий без явного разрешенного контекста;
- отключение auth, TLS, validation, logging и rate limiting;
- небезопасные shell-команды без предупреждения;
- SQL, который может привести к неограниченному destructive effect;
- инфраструктурные изменения без указания среды, риска и способа отката.

## Формат безопасного ответа

Для code/infra output желательно включать:
- краткое решение;
- сами изменения;
- ключевые риски;
- что проверить перед merge или deploy.

Для review output:
- сначала список проблем;
- затем последствия;
- затем минимально достаточные рекомендации по исправлению.

## Fail-safe правило

Если данных недостаточно для безопасного решения, агент должен:
- остановиться на более безопасном варианте;
- явно назвать недостающие факты;
- не заполнять критичные пробелы домыслами.
