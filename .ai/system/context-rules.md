# Context rules

## Назначение

Этот документ определяет правила формирования контекста для LLM, чтобы ускорять работу без роста числа ошибок и утечек.

## Основные правила

- Передавать только минимально необходимый контекст.
- Не передавать секреты, токены, ключи и реальные чувствительные данные.
- Не передавать большие файлы без предварительного сокращения.
- Использовать allow-list подход.
- Сначала включать правила, потом код, потом только нужные дополнительные материалы.

## Что включать в контекст

- файлы, напрямую относящиеся к задаче;
- сигнатуры и контракты, от которых зависит решение;
- краткие фрагменты логов после редактирования и маскирования;
- только нужные `.ai`-правила по стеку и типу задачи.

## Что не включать в контекст

- `.env`, `*.pem`, `*.key`, `*.crt`, raw credentials;
- дампы БД;
- сырые логи с персональными данными;
- документы и выгрузки, содержащие банковскую тайну или коммерческую тайну, если они не обезличены и не нужны строго для задачи;
- крупные бинарные файлы;
- нерелевантные директории целиком;
- старые артефакты, которые не влияют на решение.

## Практика отбора

- Разбивать сложную задачу на шаги.
- Для каждого шага включать только свой набор файлов.
- Использовать DSL и templates для сложных запросов.
- Ограничивать размер prompt и удалять дублирующуюся информацию.
- Выделять trusted instructions отдельно от пользовательского контекста.

## Правила сокращения

- Лучше передать точный фрагмент файла, чем целый файл.
- Лучше передать summary интерфейса, чем весь модуль реализации.
- Лучше передать схему таблицы и запрос, чем дамп БД.
- Лучше передать один релевантный stack trace, чем весь лог-файл.

## Контекст для быстрого и безопасного кодинга

Чтобы агент писал быстрее и с меньшим числом ошибок, в контекст стоит включать:
- существующие интерфейсы и модели;
- текущие тесты и контракты;
- style constraints проекта;
- ограничения по runtime, SLA и production behavior;
- правила по observability и rollback, если задача operational.

## Запрещено

- передавать в модель то, что нельзя безопасно показать в логах;
- передавать в модель внутренние бизнес-показатели, условия договоров, антифрод-правила, лимиты и другие сведения коммерческой или банковской тайны без строгой необходимости;
- подменять исходный код пересказом, если важны детали поведения;
- включать conflicting instructions без явного приоритета;
- считать предыдущий output модели проверенным источником истины.
