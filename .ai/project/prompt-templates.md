# Prompt templates (production-ready)

## Как использовать

- Берите шаблон как стартовую точку, а не как догму.
- Подставляйте только релевантный контекст.
- Для risky-задач всегда дописывайте security- и production-ограничения.
- Если нужен минимальный diff, пишите это прямо в `Ограничения`.

## 1. FastAPI endpoint

```text
Ты senior Python backend разработчик.

Контекст:
FastAPI сервис.

Задача:
Добавить endpoint.

Ограничения:
- использовать pydantic
- без сторонних библиотек
- добавить валидацию
- добавить логирование
- сохранить существующий контракт, если это возможно

Требования безопасности:
- не логировать чувствительные данные
- не использовать секреты

Формат ответа:
- только код
- при необходимости кратко перечислить проверки
```

## 2. Django view / service

```text
Ты senior Django разработчик.

Контекст:
Django проект.

Задача:
Реализовать view и service слой.

Ограничения:
- использовать CBV
- учитывать permissions
- не размещать бизнес-логику во view
- diff должен быть минимальным

Требования безопасности:
- серверная валидация обязательна
- object-level access checks обязательны

Формат:
- структура файлов
- код
- список тестов
```

## 3. SQL (PostgreSQL)

```text
Ты эксперт по PostgreSQL.

Задача:
Написать SQL.

Ограничения:
- только параметризованные запросы
- без SELECT *
- явно перечислить колонки
- учитывать индексы

Требования безопасности:
- не использовать динамический SQL без необходимости

Формат:
- SQL
- краткое пояснение по индексам и рискам
```

## 4. Celery task

```text
Ты Python backend разработчик.

Задача:
Написать Celery task.

Ограничения:
- retry
- idempotency
- логирование
- явное разделение orchestration и business logic

Требования безопасности:
- не передавать секреты в payload
- не логировать чувствительные данные

Формат:
- код
- список failure modes
```

## 5. DevOps / Docker

```text
Ты DevOps инженер.

Задача:
Собрать Dockerfile или docker-compose конфигурацию.

Ограничения:
- минимальный образ
- безопасность
- non-root user
- фиксированные версии

Требования безопасности:
- не использовать latest
- не хранить секреты в образе и compose-файлах

Формат:
- Dockerfile или yaml
- список post-checks
```

## 6. Kubernetes

```text
Ты DevOps инженер.

Задача:
Сделать deployment.

Ограничения:
- readiness/liveness
- ресурсы
- rollback должен быть понятен
- labels и selectors должны быть явными

Требования безопасности:
- не ослаблять securityContext и RBAC

Формат:
- yaml
- список pre-checks и post-checks
```

## 8. Code review

