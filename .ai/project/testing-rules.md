# Testing rules

## Назначение

Этот документ задает правила тестирования для кода и сценариев, сгенерированных или измененных агентом.

## Основные требования

- Для новой бизнес-логики добавлять unit tests.
- Для интеграций добавлять integration tests или contract checks.
- Для критичных финансовых сценариев предусматривать negative tests.
- Для async logic проверять retry, timeout и idempotency.
- Для bugfix по возможности добавлять regression test.

## Что проверять

- корректность бизнес-результата;
- безопасность входных данных;
- граничные условия;
- обработку ошибок;
- повторное выполнение;
- race conditions там, где это применимо;
- behavior under partial failures.

## Минимальный набор для production-safe изменения

- happy path;
- минимум один negative path;
- проверка contract/shape результата;
- проверка логики при повторном вызове, если операция mutating;
- проверка observability, если изменение влияет на логи, статусы или retries.

## Для API и сервисов

- проверять статусы ошибок;
- проверять валидацию request payload;
- проверять сериализацию ответа;
- проверять, что sensitive fields не утекают наружу.

## Для БД и async

- проверять transaction boundaries;
- проверять идемпотентность при повторной доставке;
- проверять behavior при timeout и retry;
- проверять, что дубль не приводит к повторному side effect.

## Для LLM-сценариев

- Проверять output модели до merge.
- Не принимать код без ручного review.
- Для сложных prompt-сценариев использовать regression suite.
- При изменении prompt templates проверять типовые сценарии на регрессию.

## Если тесты не добавлены

Нужно явно указать:
- что не покрыто;
- почему тесты не добавлены;
- какой риск остается до merge или deploy.
