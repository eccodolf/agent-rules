# Review checklist

## Перед merge

- код читаемый и достаточно локальный по diff;
- есть тесты или явно описано, почему их нет;
- нет секретов и чувствительных данных;
- соблюдены правила безопасности;
- есть логирование там, где оно действительно нужно;
- есть обработка ошибок и failure paths;
- не сломан существующий контракт без явного указания.

## Для кода

- входные данные валидируются;
- side effects явные;
- нет silent failure и swallow exceptions;
- нет небезопасных SQL или shell-паттернов;
- учтены timeout, retry и duplicate execution, если это релевантно.

## Для API

- статусы и ошибки предсказуемы;
- response не раскрывает внутренние детали;
- auth/authz проверяются в нужной точке;
- чувствительные поля не утекают наружу.

## Для инфраструктуры

- есть rollback;
- есть pre-checks и post-checks;
- версии зафиксированы;
- нет ослабления security controls без причины;
- изменение воспроизводимо и audit-friendly.

## Для банковских сценариев

- есть audit;
- есть idempotency;
- есть rollback или compensating strategy;
- есть duplicate protection;
- права и подтверждения проверяются явно.
