# Logging rules

## Назначение

Правила логирования для кода и operational-сценариев, чтобы ускорять диагностику и не создавать новые security-риски.

## Основные требования

- Логировать ключевые действия и переходы состояния.
- Не логировать секреты, токены и необработанные персональные данные.
- Использовать structured logging там, где это уже принято в проекте.
- Для запросов и фоновых задач использовать correlation id и trace id.

## Что должно попадать в логи

- тип операции;
- outcome: success/failure/retry;
- безопасные идентификаторы сущностей;
- причина ошибки без утечки чувствительных деталей;
- request id, job id или trace id;
- decision points для critical flows.

## Что не должно попадать в логи

- секреты;
- платежные данные и персональные реквизиты в явном виде;
- полный request/response body без необходимости;
- raw SQL с чувствительными параметрами;
- stack trace в пользовательский output.

## Практика

- Маскировать чувствительные поля.
- Логировать enough context for debugging, но не весь payload.
- Делать логи пригодными для алертов и расследования.
- Для retry и async явно логировать номер попытки и причину повтора.

## Production-safe правила

- Логи должны помогать отличить transient failure от logical bug.
- В financial flows должны быть audit-friendly события.
- Не использовать noisy logging, который скрывает реальные ошибки.
- Не писать debug-логи в production по умолчанию.
