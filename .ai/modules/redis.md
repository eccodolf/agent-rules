# Redis module rules

## Назначение

Этот документ задает правила для задач, связанных с Redis: cache, rate limiting, distributed locks, очереди, временное состояние и coordination patterns.

## Основные правила

- Явно определять, для чего используется Redis: cache, lock, queue, ephemeral state, rate limit.
- Не использовать Redis как primary source of truth для критичных банковских данных.
- Любое ключевое пространство должно иметь понятный namespace.
- Для временных данных задавать TTL.
- Для кэширования указывать стратегию invalidation.
- Для блокировок явно описывать timeout, renewal и failure mode.
- Для финансовых операций не полагаться только на Redis lock как на единственный механизм защиты от дублей.

## Безопасность

- Не хранить в Redis секреты, приватные ключи и долговременные токены.
- Не хранить необезличенные чувствительные данные без явной необходимости и компенсационных мер.
- Ограничивать сетевой доступ к Redis.
- Использовать auth/TLS там, где это поддерживается целевой инфраструктурой.
- Не логировать значения ключей, если они содержат чувствительные идентификаторы.

## Надежность для production

- Для кэша всегда думать о поведении при cache miss и Redis outage.
- Для distributed lock учитывать истечение TTL, split brain и потерю владельца.
- Для rate limiting понимать поведение при burst и degraded mode.
- Для async coordination не смешивать техническое состояние и бизнес-состояние без причины.

## Логирование

- Логировать cache hit/miss, lock acquire/release failures и rate-limit decisions без утечки значений ключей.
- Не логировать полные значения ключей, если они содержат идентификаторы клиентов, операций или иные чувствительные данные.
- Для Redis outage и timeout логировать деградацию сервиса и fallback behavior.
- Для distributed lock конфликтов логирование должно позволять расследовать lost ownership и duplicate execution.

## Архитектурные рекомендации

- Ключи именовать в формате `service:domain:entity:id[:suffix]`.
- Для счетчиков и rate limits использовать atomic operations.
- Для кэша описывать policy: cache-aside, write-through или explicit invalidation.
- Для distributed lock учитывать split brain и истечение TTL.
- Для Celery и очередей не смешивать operational state и бизнес-состояние.

## Что запрещено

- Хранить в Redis единственную копию финансово значимого состояния.
- Использовать бесконечный TTL без причины.
- Создавать ключи без namespace.
- Реализовывать идемпотентность только через Redis без дополнительной серверной проверки.
- Полагаться на Redis как на гарантию консистентности там, где нужна транзакционная модель.
