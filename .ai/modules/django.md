# Django module rules

## Основные правила

- Использовать явное разделение `views`, `services`, `repositories/selectors` там, где это уместно.
- Не размещать сложную бизнес-логику в `views`, `forms`, `serializers` и `admin`.
- Для ORM-запросов использовать `select_related` и `prefetch_related` осознанно.
- Не допускать N+1 запросов.
- Для прав доступа использовать permissions, mixins или policy-слой.
- Для настроек использовать переменные окружения и централизованный конфиг.
- Для миграций соблюдать обратимость и предсказуемость.

## Безопасность

- Проверять доступ к объектам, а не только к endpoint.
- Не отключать CSRF без явной причины и компенсационных мер.
- Не использовать небезопасный `mark_safe` без строгого контроля.
- Не доверять данным из request без валидации.
- Не включать `DEBUG` в production.

## Надежность для production

- Избегать скрытых side effects в `save()`, signals и middleware.
- Для state changes использовать явные сервисные методы и `transaction.atomic`, если это нужно.
- Не смешивать read- и write-логику в хаотичном виде внутри view.
- Учитывать конкурентные обновления и повторные вызовы для mutating actions.

## Что запрещено

- Бизнес-логика в signals без необходимости.
- Скрытые side effect в `model.save()`.
- Доступ к секретам напрямую из кода.
- Сырые SQL-запросы без крайней необходимости и review.
- "Удобные" admin actions без проверки прав и аудита.
