# Django module rules

## Основные правила

- Использовать явное разделение `views`, `services`, `repositories/selectors` там, где это уместно.
- Не размещать сложную бизнес-логику в `views`, `forms`, `serializers` и `admin`.
- Для ORM-запросов использовать `select_related` и `prefetch_related` осознанно.
- Не допускать N+1 запросов.
- Для прав доступа использовать permissions / mixins / policy-слой.
- Для настроек использовать переменные окружения и централизованный конфиг.
- Для миграций соблюдать обратимость и предсказуемость.

## Безопасность

- Проверять доступ к объектам, а не только к endpoint.
- Не отключать CSRF без явной причины и компенсационных мер.
- Не использовать небезопасный `mark_safe` без строгого контроля.
- Не доверять данным из request без валидации.
- Не включать DEBUG в production.

## Что запрещено

- Бизнес-логика в signals без необходимости.
- Скрытые side effect в model save().
- Доступ к секретам напрямую из кода.
- Сырые SQL-запросы без крайней необходимости и review.

## Примеры промптов

### Простой

```text
Ты senior Django разработчик.

Контекст:
Есть Django проект.

Задача:
Создай CBV для отображения списка клиентов.

Ограничения:
- использовать LoginRequiredMixin
- добавить пагинацию

Формат ответа:
- код view
- краткое пояснение
```

### Средний

```text
Ты senior Django backend разработчик.

Контекст:
Есть приложение customers. Нужно добавить создание клиента через Django FormView и service слой.

Задача:
Подготовь структуру решения и код.

Ограничения:
- не размещать бизнес-логику во view
- выполнить серверную валидацию
- предусмотреть audit logging
- не использовать signals

Формат ответа:
1. структура файлов
2. код по файлам
3. список проверок
```

### Сложный

```text
Ты Django архитектор для банковской системы.

Контекст:
Есть внутренний портал на Django. Нужно реализовать изменение статуса заявки с разграничением прав, аудитом и защитой от повторной обработки.

Задача:
Спроектируй и опиши реализацию.

Ограничения:
- использовать transaction.atomic
- проверка прав на объект обязательна
- действия должны писаться в audit trail
- изменение должно быть идемпотентным
- ошибки не должны раскрывать внутреннюю структуру

Формат ответа:
1. архитектурный подход
2. код view/service/model changes
3. тестовые сценарии
4. риски безопасности
```
