# Celery module rules

## Основные правила

- Все задачи должны быть идемпотентными.
- Использовать retry с backoff и ограничением числа попыток.
- Ограничивать время выполнения задач.
- Явно разделять task orchestration и бизнес-логику.
- Payload задач должен быть компактным и безопасным.

## Безопасность

- Не передавать секреты в payload задачи.
- Не логировать чувствительные данные.
- Не полагаться на очередь как на security boundary.

## Надежность для production

- Учитывать duplicate delivery и at-least-once semantics.
- Не делать необратимый side effect без duplicate protection.
- Для chains/groups/chords явно описывать failure behavior.
- Для retry-safe операций разделять transient и permanent failures.
- Для задач, влияющих на деньги и статус, иметь audit trail вне брокера.

## Что запрещено

- неидемпотентные задачи без защитного ключа или проверки состояния;
- бесконечные retries без лимита и без анализа ошибки;
- тяжелая бизнес-логика без таймаутов и контроля ресурсов;
- использование Celery result backend как основного источника бизнес-истины.
