# FastAPI module rules

## Назначение

Этот документ задает правила для задач, связанных с FastAPI.

## Основные правила

- Использовать явные схемы `pydantic` для входа и выхода.
- Не возвращать ORM-объекты напрямую из endpoint.
- Валидировать все входные параметры.
- Для ошибок использовать предсказуемые HTTP-коды и структурированные ответы.
- Разделять router, service, repository и schema слои.
- Не размещать бизнес-логику внутри endpoint-функций.
- Для зависимостей использовать `Depends`.
- Любые внешние вызовы должны иметь таймауты и обработку ошибок.
- Логирование не должно содержать секреты, токены, персональные данные и полные payload с чувствительными полями.

## Безопасность

- Проверять аутентификацию и авторизацию явно.
- Не использовать debug-конфигурации в production.
- Ограничивать выдачу внутренних деталей ошибок.
- Для загрузки файлов задавать ограничения размера и типа.
- Проверять пользовательский ввод перед использованием в SQL, filesystem и внешних запросах.

## Надежность для production

- Для mutating endpoint учитывать идемпотентность.
- Для long-running operations отделять HTTP-прием запроса от фонового выполнения.
- Для внешних интеграций учитывать timeout, retry и circuit-breaker pattern, если он есть в проекте.
- Для ошибок различать client error, business error и infrastructure error.
- Не блокировать event loop CPU-heavy задачами и синхронным I/O без причины.

## Архитектурные рекомендации

- `api/routers/` — только HTTP-слой.
- `services/` — бизнес-логика.
- `repositories/` — доступ к данным.
- `schemas/` — входные и выходные модели.
- `dependencies/` — DI и security dependencies.

## Что запрещено

- Писать SQL прямо в endpoint.
- Смешивать transport-model и domain-model без необходимости.
- Использовать глобальное состояние без явного контроля.
- Возвращать stack trace клиенту.
- Прятать side effects внутри dependency без явной необходимости.
