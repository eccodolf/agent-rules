# Vault module rules

## Назначение

Этот документ задает правила работы с секретами через Vault (HashiCorp Vault или аналог) в системах банковского уровня.

## Основные принципы

- Все секреты хранятся только в Vault.
- Приложения не должны содержать секреты в коде или конфигурациях.
- Использовать динамические секреты, где это возможно.
- Секреты должны иметь TTL и ротацию.
- Доступ к секретам должен быть минимально необходимым (least privilege).

## Безопасность

- Никогда не логировать секреты.
- Не выводить секреты в ответах API.
- Ограничивать scope токенов Vault.
- Использовать short-lived токены.
- Контролировать доступ через политики Vault.

## Архитектурные рекомендации

- Использовать AppRole/Kubernetes auth вместо статических токенов.
- Разделять секреты по environment (dev/test/prod).
- Использовать path-based разграничение доступа.
- Использовать transit engine для шифрования данных.

## Что запрещено

- Хранить секреты в `.env` для production.
- Использовать один общий токен для всех сервисов.
- Передавать секреты через LLM.

## Примеры промптов

### Простой

```text
Объясни, как хранить секреты в Vault.
```

### Средний

```text
Ты DevOps инженер.

Контекст:
Есть сервис на FastAPI.

Задача:
Настроить получение секретов из Vault.

Ограничения:
- использовать AppRole
- не хранить секреты в env

Формат:
- схема
- пример кода
```

### Сложный

```text
Ты архитектор безопасности.

Контекст:
Банковская система.

Задача:
Спроектировать управление секретами через Vault.

Ограничения:
- dynamic secrets
- rotation
- audit

Формат:
- архитектура
- политики
- риски
```
