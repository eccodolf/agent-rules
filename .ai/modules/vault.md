# Vault module rules

## Назначение

Этот документ задает правила работы с секретами через Vault в системах production- и banking-уровня.

## Основные принципы

- Все секреты хранятся только в Vault или утвержденном secret manager.
- Приложения не должны содержать секреты в коде или конфигурациях.
- Использовать динамические секреты там, где это возможно.
- Секреты должны иметь TTL и ротацию.
- Доступ к секретам должен быть минимально необходимым.

## Безопасность

- Никогда не логировать секреты.
- Не выводить секреты в ответах API и output модели.
- Ограничивать scope токенов Vault.
- Использовать short-lived токены.
- Контролировать доступ через политики Vault.

## Надежность для production

- Ошибки доступа к Vault должны быть обработаны явно.
- Сервис должен деградировать безопасно, а не переходить на insecure default.
- Secret retrieval path должен быть наблюдаемым без раскрытия значений.
- Нельзя строить critical startup logic так, чтобы сервис бесконечно падал без понятной диагностики.

## Логирование

- Логировать факт обращения к секрету, используемый path/class of secret и outcome без раскрытия значения.
- Не логировать token values, secret payload, lease contents и расшифрованные данные.
- Для ошибок auth, lease renewal и access denied логировать причину и технический контекст без утечки чувствительных деталей.
- Логи должны позволять понять, была ли проблема в политике доступа, доступности Vault или lifecycle секрета.

## Архитектурные рекомендации

- Использовать AppRole, Kubernetes auth или другой machine-to-machine auth вместо статических токенов.
- Разделять секреты по environment.
- Использовать path-based разграничение доступа.
- Использовать transit engine для шифрования данных, когда это уместно.
- Разделять секреты приложения, инфраструктуры и CI/CD.

## Что запрещено

- Хранить секреты в `.env` для production.
- Использовать один общий токен для всех сервисов.
- Передавать секреты через LLM.
- Подменять штатный secret flow временными значениями "на время отладки".
